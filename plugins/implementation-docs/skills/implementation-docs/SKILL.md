# Implementation Docs 스킬

## 역할

구현된 코드를 바탕으로 **개발자가 바로 쓸 수 있는 가이드 문서**를 작성한다.
API 분석 문서(Confluence 복사본)가 아니라, **이 프로젝트의 코드 기준**으로 작성한다.

**핵심 원칙**: "어떻게 구현했는가"가 아니라 "어떻게 사용하는가"를 중심으로 쓴다.

---

## Phase 1: 컨텍스트 수집

### 1-1. 계획/합의/도메인 문서 읽기

- `docs/.consensus/` — 기술 결정 사항
- `docs/.domain/` — 포트 인터페이스, 비즈니스 규칙
- `docs/.plan/` — 구현 태스크, 파일 목록

### 1-2. 기존 docs 구조 파악

`docs/` 디렉토리 탐색 → 기존 가이드 파일 1-2개 읽어 패턴 파악.
기존 스타일(헤딩 레벨, 다이어그램 형식, 섹션 구조)을 동일하게 적용.

### 1-3. 구현 코드 읽기

문서화 대상 클래스 파일들을 직접 읽는다:
- 포트 인터페이스 (메서드 시그니처, 파라미터, 반환 타입)
- 설정 클래스 (`@ConfigurationProperties` 필드)
- 주요 구현체 (핵심 로직, 분기 조건)

코드를 직접 읽지 않고 문서를 쓰면 틀린 정보가 들어간다. 반드시 읽는다.

---

## Phase 2: 문서 범위 결정

`$ARGUMENTS`가 있으면 해당 내용을 문서화 대상으로 사용.
없으면: plan 문서에서 구현된 태스크 목록 기준으로 대상 파악.

여러 독립 기능이 섞여 있으면 **파일을 분리**한다:
- 기능별 1개 파일 원칙 (예: `chase-reporting-client.md`, `cybersource-reporting-client.md`)
- 밀접하게 연관된 기능은 1개 파일로 합침

---

## Phase 3: 문서 작성

### 저장 위치

`docs/{도메인}/{기능명}.md`

- 디렉토리 없으면 생성
- 기존 docs 구조 참고해서 적절한 위치 결정 (예: `docs/pg/`, `docs/batch/`)
- 파일명은 `kebab-case`

### 문서 구조

아래 섹션을 기본으로 하되, 내용이 없는 섹션은 생략한다.

```markdown
# {기능명} 구현 가이드

> {JIRA_KEY} | {한 줄 설명}

---

## 1. 개요

{이 컴포넌트가 해결하는 문제. 범위(In Scope) 포함.}

---

## 2. 아키텍처

{레이어 구조 다이어그램 (텍스트 또는 mermaid). 호출 흐름 포함.}

---

## 3. 설정

{application.yml 예시. 환경변수 명시. 환경별 차이 있으면 표로 정리.}

---

## 4. 인증

{인증 메커니즘 설명. 토큰 캐싱, 서명 방식 등 동작 원리.}
(인증이 단순하면 생략)

---

## 5. 포트 인터페이스

{코드에서 읽어온 실제 인터페이스 정의 + 결과 DTO 구조.}

---

## 6. 핵심 동작

{주요 비즈니스 로직, 분기 조건, 상태 흐름 등.}
{배치 Reader 사용 예시 코드 포함.}

---

## 7. 주의사항

| 항목 | 내용 |
|------|------|
| {주의할 점} | {구체적인 내용} |
```

### 작성 원칙

**코드 기반**: 인터페이스, 설정 필드, 메서드 시그니처는 실제 코드에서 읽어온 것을 그대로 쓴다. 추측하지 않는다.

**사용자 관점**: "어떻게 구현했는지"가 아니라 "배치 Job에서 어떻게 호출하는지"를 중심으로 쓴다.

**예시 코드**: 핵심 사용 패턴을 Kotlin 코드 블록으로 제공한다. 컴파일 되는 코드여야 한다.

**주의사항**: 운영 환경에서 자주 실수하거나 빠뜨리는 것을 표로 정리한다.

**금지 패턴:**
- "다양한 방법이 있습니다" 같은 모호한 표현
- Confluence 문서 복사 붙여넣기 (이미 있는 내용 중복 불필요)
- 구현 히스토리나 과정 설명 ("처음에는 X로 했다가...")
- TBD, 미정 항목

---

## Phase 4: 자가 검토

문서 저장 전 확인:

1. **코드 일치**: 인터페이스 시그니처, 설정 필드명이 실제 코드와 일치하는가?
2. **예시 코드**: 실제로 컴파일 되는가? 존재하지 않는 클래스/메서드를 쓰지 않았는가?
3. **커버리지**: plan의 주요 태스크가 문서에 반영됐는가?
4. **사용자 관점**: 이 문서만 보고 배치 Job에서 이 컴포넌트를 쓸 수 있는가?

---

## Phase 5: 완료 보고

```
## 문서 작성 완료 ✅

저장 위치:
- docs/{경로}/{파일1}.md
- docs/{경로}/{파일2}.md (있는 경우)

### 주요 섹션
- 아키텍처: {포트/어댑터 구조 등}
- 설정: {프로퍼티 목록}
- 사용 예시: {배치 Reader 등}

### 다음 단계

**A. git add** — 문서를 현재 브랜치에 stage
**B. 종료**
```

사용자가 A를 선택하면 문서 파일을 `git add`한다.

---

## 출력 언어

**항상 한글로 작성한다.** 클래스명, 메서드명, 파일 경로, 기술 용어는 원어 그대로 사용한다.
