# Implementation Review 스킬

## 역할

구현이 계획대로 이루어졌는지를 문서 기반으로 검증한다. 코드 품질 리뷰가 아니라 **"합의한 것을 구현했는가"** 가 핵심 질문이다. 평가 기준을 먼저 생성하고, 기준별 점수를 매긴 뒤, 미달 항목을 수정하여 전체 통과까지 반복한다.

---

## Phase 1: 문서 로드

다음 순서로 컨텍스트 문서를 전부 읽는다.

1. `docs/.consensus/` — 합의 결정 사항
2. `docs/.domain/` — 도메인 모델, 비즈니스 규칙, Aggregate 구조
3. `docs/.plan/` — 구현 태스크, 완료 기준, 테스트 계획

없는 문서는 건너뛴다. 세 가지 모두 없으면 `$ARGUMENTS`에서 컨텍스트 파악, 그래도 없으면 git diff와 현재 코드만으로 일반 리뷰 수행.

---

## Phase 2: 평가 기준(루브릭) 생성

로드한 문서를 분석하여 **이 구현에 특화된 평가 기준**을 생성한다. 고정 기준이 아니라 문서 내용에서 추출한 구체적 항목이어야 한다.

평가 기준은 5개 카테고리로 구성하고, 각 카테고리 내 세부 항목을 문서에서 추출한다.

### 카테고리 구조

```
[C1] 계획 적합성 (Plan Compliance)          — 가중치 30%
  → plan 문서의 각 태스크 완료 기준이 구현됐는가

[C2] 비즈니스 규칙 반영 (Business Rules)    — 가중치 25%
  → domain 문서의 비즈니스 규칙이 코드에 구현됐는가
  → Aggregate 불변 조건이 Entity 내부에 캡슐화됐는가

[C3] 합의 사항 반영 (Consensus Alignment)   — 가중치 20%
  → consensus 문서의 결정이 구현에 반영됐는가
  → 미결 사항이 있으면 적절히 처리됐는가

[C4] 테스트 커버리지 (Test Coverage)        — 가중치 15%
  → plan의 테스트 계획 항목이 실제로 구현됐는가 (작성된 테스트 수가 아닌 plan 항목 기준으로 계산)
  → 비즈니스 규칙마다 테스트가 존재하는가
  → "MockWebServer 필요", "환경 구성 필요" 등의 이유로 테스트를 건너뛰는 것은 허용하지 않음
    — 실제로 테스트를 작성하거나, 사용자가 명시적으로 제외를 승인해야만 ⚠️/❌로 처리 가능

[C5] 코딩 컨벤션 (Code Conventions)         — 가중치 10%
  → CLAUDE.md 또는 프로젝트 컨벤션 준수 여부
```

### 세부 항목 점수

각 세부 항목은 3단계 평가:
- ✅ **통과 (1.0점)**: 명확히 구현됨
- ⚠️ **부분 (0.5점)**: 일부 구현됐으나 누락/미흡
- ❌ **실패 (0.0점)**: 미구현 또는 잘못 구현됨

**통과 기준: 전체 가중 점수 85% 이상**

---

## Phase 3: 평가 실행

생성한 루브릭을 기준으로 현재 구현 코드를 평가한다.

각 세부 항목에 대해:
1. 관련 코드 파일을 직접 읽는다
2. 항목 기준과 대조한다
3. 점수와 근거를 기록한다

### C4 테스트 커버리지 평가 규칙

**⚠️ 절대 금지**: "MockWebServer 필요", "통합 테스트라 로컬 환경 필요", "별도 환경 구성 필요" 등의 이유로 테스트 작성을 건너뛰고 점수를 낮춰서 통과 판정하는 것.

**계산 방식**: plan 문서의 테스트 계획 테이블 각 행을 1개 항목으로 센다.
- 작성된 테스트 케이스 수가 아니라 **plan 항목 기준**으로 계산
- 예: 5개 테스트 케이스가 plan 항목 3개를 커버하면 → 3/8, 37.5% (5/8 아님)

**불가피한 제외 처리**:
- 실제 외부 API 키/인증서가 없어 통합테스트가 불가능한 경우 → 사용자에게 명시적 승인 요청 후 제외 가능
- 그 외 MockWebServer, mock WebClient, MockK 등으로 대체 가능한 경우 → 반드시 작성

### 평가 결과 출력 형식

```
## 구현 검증 결과 (Round {N})

### 점수 요약
| 카테고리 | 점수 | 가중 점수 |
|---------|------|---------|
| C1. 계획 적합성 | {X}/{Y} ({%}) | {가중치 적용} |
| C2. 비즈니스 규칙 | {X}/{Y} ({%}) | |
| C3. 합의 사항 | {X}/{Y} ({%}) | |
| C4. 테스트 커버리지 | {X}/{Y} ({%}) | |
| C5. 코딩 컨벤션 | {X}/{Y} ({%}) | |
| **총점** | | **{최종%}** |

**{통과 ✅ / 미달 ❌}** (기준: 85%)

### 세부 항목

#### ✅ 통과
- [C1] {태스크명}: {근거}
- [C2] {규칙명}: {코드 위치}

#### ❌ 실패 / ⚠️ 부분
- [C2] {규칙명}: {문제점} → {수정 방향}
- [C4] {테스트 항목}: {누락 이유} → {추가해야 할 테스트}
```

## Phase 4: 수정 루프

총점이 85% 미만이면 자동으로 수정 루프에 진입한다.

### 수정 우선순위

1. ❌ 실패 항목 (0.0점) 먼저 — 가중치 높은 카테고리 순
2. ⚠️ 부분 항목 (0.5점) — 완전 통과로 올릴 수 있는 것 우선
3. 동점이면 C1(계획 적합성) → C2(비즈니스 규칙) → C3 → C4 → C5 순

**⚠️ 절대 금지**: 85%를 간신히 넘겼다는 이유로 C4(테스트) 미달 항목을 그냥 두는 것.
총점이 85%를 넘더라도 C4 항목에 ❌/⚠️가 남아 있으면 **반드시 수정 루프를 계속**한다.
테스트는 점수를 맞추기 위한 수단이 아니라 구현의 정확성을 보장하는 필수 요소다.

### 수정 방식

각 실패 항목에 대해:
1. 문제의 근본 원인 파악 (구현 누락? 잘못된 위치? 규칙 미반영?)
2. 최소한의 수정만 적용 (주변 코드 리팩터링 금지)
3. 수정 후 해당 카테고리만 재평가
4. 모든 수정 완료 후 전체 점수 재산출

### 루프 종료 조건

| 조건 | 액션 |
|------|------|
| 총점 ≥ 85% | 통과 선언, Phase 5로 |
| 3회 시도 후에도 특정 항목 실패 | 사용자에게 보고, 결정 요청 |
| 수정이 다른 항목 점수를 낮춤 | 사용자에게 트레이드오프 보고 |

사용자 개입 없이 계속 시도하되, 막히면 투명하게 보고한다.

