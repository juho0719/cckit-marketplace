# Domain Definition 스킬

## 역할

구현 전에 DDD 방법론으로 도메인을 분석하고 문서화한다. 이 문서는 계획(plan) 작성과 구현의 기준이 되어, AI 에이전트가 잘못된 방향으로 구현할 확률을 줄인다.

**핵심 전제**: 도메인이 정의되지 않은 상태로 계획을 세우면, 계획 자체가 흔들린다. 도메인 정의 → 계획 → 구현 순서를 지킨다.

---

## Phase 1: 입력 수집

`$ARGUMENTS`가 있으면 해당 내용을 컨텍스트로 사용한다.

입력이 없거나 부족하면 다음 순서로 컨텍스트를 보충한다:
1. 직전 consensus 스킬 결과가 대화 내에 있으면 그것을 사용
2. 현재 프로젝트의 CLAUDE.md, 관련 기존 코드를 탐색하여 패턴 파악
3. 그래도 모호하면 핵심 질문 하나만: "어떤 도메인을 정의할까요? 핵심 비즈니스 행위를 한 문장으로 설명해주세요."

---

## Phase 2: 도메인 분석

다음 항목을 순서대로 분석한다. 각 항목에서 불명확한 부분은 추론하되, 추론임을 `[추론]` 태그로 표시한다. 확실히 알 수 없는 항목은 `N/A`로 표기한다.

### 2-1. Bounded Context 식별

이 도메인이 어느 경계 안에 있는지 정의한다.
- 이 도메인이 책임지는 것 (In scope)
- 이 도메인이 책임지지 않는 것 (Out of scope)
- 인접 도메인과의 관계

### 2-2. Aggregate 설계

- **Aggregate Root**: 외부에서 진입하는 유일한 엔트리포인트 엔티티
- **하위 Entity**: Aggregate Root 내부에서만 참조 가능한 엔티티
- **Value Object**: 식별자 없이 값으로만 의미를 갖는 객체
- **불변 조건(Invariant)**: Aggregate가 항상 보장해야 하는 규칙

Aggregate 경계 판단 기준:
- 트랜잭션 일관성이 필요한 범위를 하나의 Aggregate로 묶는다
- Aggregate가 너무 크면 동시성 충돌 증가, 너무 작으면 트랜잭션 관리 복잡

### 2-3. Domain Event

이 도메인에서 발생하는 의미 있는 사건을 정의한다. 다른 도메인이 반응해야 하는 사건이 있을 때 특히 중요하다.
- 형식: `{주어}{과거형동사}` (예: `PaymentCompleted`, `CancellationApproved`)
- 이벤트가 없는 단순 CRUD 도메인이면 `N/A`

### 2-4. Repository 인터페이스

도메인이 데이터 접근을 위해 요구하는 계약을 정의한다. 구현 기술(JPA, QueryDSL 등)은 명시하지 않는다.
- 기본 CRUD 외에 도메인이 필요로 하는 특수 조회/저장 메서드
- 복수 결과 메서드는 `findAllBy~` 형식
- 단건은 `findBy~` 형식

### 2-5. Domain Service / Policy

엔티티 하나에 속하기 어려운 도메인 로직을 명시한다.
- 여러 Aggregate를 조합해야 하는 비즈니스 규칙
- 외부 도메인과 협력이 필요한 정책

### 2-6. 외부 시스템 의존성

이 도메인이 의존하는 외부 시스템을 정의한다. 포트/어댑터 패턴이 필요한 지점을 명확히 한다.
- 외부 API, 메시지 브로커, 레거시 시스템 등
- 의존 방향: 이 도메인 → 외부 (능동) vs 외부 → 이 도메인 (수동)
- 해당 없으면 `N/A`

### 2-7. 비즈니스 규칙

도메인의 핵심 제약 조건과 정책을 열거한다. 구현보다 비즈니스 관점으로 서술한다.
- 형식: "~인 경우, ~해야 한다 / ~할 수 없다"
- 예외 조건, 상태 전이 규칙 포함

## Phase 3: 문서 생성

분석 결과를 `docs/.domain/{도메인명}.md`에 저장한다.

`docs/.domain/` 디렉토리가 없으면 생성한다.

도메인명은 `kebab-case`로 지정한다 (예: `payment-cancellation`, `settlement-account`).

### 문서 템플릿

```markdown
# {도메인명} 도메인 정의

> 작성일: {날짜} | 기반: {consensus 결과 또는 직접 분석}

## 한 줄 요약

{이 도메인이 해결하는 핵심 문제를 한 문장으로}

## Bounded Context

### In Scope
- {이 도메인이 책임지는 것}

### Out of Scope
- {이 도메인이 책임지지 않는 것}

### 인접 도메인
| 도메인 | 관계 | 연동 방식 |
|--------|------|---------|
| {도메인명} | {upstream/downstream} | {동기/이벤트/공유DB} |

```kotlin

## Phase 4: 완료 보고 및 다음 단계 선택

문서 저장 후 다음을 출력한다:

```
## 도메인 정의 완료 ✅

저장 위치: docs/.domain/{도메인명}.md

### 핵심 요약
- Aggregate Root: {이름}
- 핵심 비즈니스 규칙: {개수}개
- Domain Event: {개수}개 / 외부 의존: {개수}개
- 미결/추론 항목: {개수}개 → 구현 전 검토 필요

### 다음 단계

**A. /writing-plan** ← 추천
도메인 정의를 바탕으로 구현 계획을 작성합니다.

**B. 종료** (계획은 나중에 작성)
지금은 도메인 정의까지만 진행합니다.
```

사용자가 A를 선택하면 `/writing-plan`을 즉시 실행한다. B이거나 "나중에"라고 하면 종료한다.

## 출력 언어

**항상 한글로 작성한다.** 클래스명, 메서드명, 기술 용어(Entity, Repository, Aggregate 등)는 원어 그대로 사용한다.
