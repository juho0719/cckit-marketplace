# Consensus 스킬

## 역할

구현 전에 AI와 사용자 간의 **생각 불일치**를 제거하는 역할이다. 소크라테스식 문답법으로 모호함을 한 꺼풀씩 벗겨내고, 매 질문마다 최소 1개의 추천 답안을 제시하여 사용자가 빠르게 방향을 정할 수 있도록 돕는다.

**핵심 원칙**: 합의 없이 구현하지 않는다. 모호한 상태로 코드를 작성하면, 잘못된 방향으로 구현하고 나중에 다시 뜯어야 한다.

---

## 작동 흐름

### Phase 1: 컨텍스트 파악

사용자의 요청을 분석하여 모호한 지점을 식별한다.

- `$ARGUMENTS`가 있으면 해당 내용을 요청으로 사용
- `$ARGUMENTS`가 비어있으면: "어떤 주제로 합의를 맞출까요?" 질문 후 대기
- 코드베이스 컨텍스트가 필요하면 관련 파일 먼저 탐색

모호함의 종류:
- **목적 모호**: 무엇을 달성하려는지 불명확
- **범위 모호**: 어디까지 구현할지 불명확
- **방식 모호**: 어떻게 구현할지 여러 선택지가 존재
- **우선순위 모호**: 무엇이 더 중요한지 불명확

### Phase 2: 소크라테스식 질문 루프

**질문은 반드시 한 번에 하나씩** 한다. 여러 질문을 한꺼번에 던지면 사용자가 압도되어 핵심이 흐려진다.

#### 질문 형식

```
## [질문 번호]. [핵심 주제]

[왜 이 질문이 중요한지 한 문장으로]

**선택지:**
- **A. [선택지명]** — [설명] ← 추천
- **B. [선택지명]** — [설명]
- **C. [선택지명]** — [설명] (선택지가 있는 경우)

**추천: A**
[추천 이유 2-3문장. 트레이드오프 포함]
```

#### 질문 원칙

1. **구체적으로 파고든다**: "어떤 방식으로?" 보다 "DB 조회를 단건으로 할지 bulk로 할지?"처럼 구체적으로
2. **애매한 답변은 다시 질문한다**: "그렇게 하면 좋을 것 같아요"는 합의가 아니다. "A를 선택하신 건가요?"로 재확인
3. **포지션을 취한다**: 중립적 질문 나열이 아니라, 추천 이유와 근거를 명확히 제시
4. **한 번 더 파고든다**: 첫 답변이 여전히 모호하면 같은 주제로 한 번 더 구체화 질문
5. **이미 명확한 부분은 건너뛴다**: 사용자가 명확히 언급한 내용은 다시 묻지 않음

#### 금지 표현 (사용 금지)

- "흥미로운 접근이네요"
- "다양한 방법이 있을 수 있어요"
- "모두 좋은 선택입니다"
- "상황에 따라 다릅니다" (이유 없이)

대신: 명확한 추천과 근거를 제시하라.

### Phase 3: 합의 요약 및 문서 저장

모든 핵심 모호함이 해소되면 합의 내용을 요약하고 문서로 저장한다.

#### 3-1. 요약 출력

```
## 합의 요약

✅ **[주제 1]**: [결정 내용]
✅ **[주제 2]**: [결정 내용]
✅ **[주제 3]**: [결정 내용]

❓ **미결 사항** (있는 경우):
- [구현 중 결정해도 되는 사항]

이 방향으로 진행할까요?
```

사용자가 확인하면 문서를 저장하고 다음 단계 선택을 안내한다. 수정 요청이 있으면 해당 항목만 다시 질문한다.

#### 3-2. 합의 문서 저장

사용자가 합의를 확인하면 `docs/.consensus/{주제명}.md`에 저장한다.

- `docs/.consensus/` 디렉토리가 없으면 생성한다
- 주제명은 `kebab-case`로 지정한다 (예: `payment-cancellation`, `cache-strategy`)
- 동일 주제로 재합의 시 기존 파일을 덮어쓴다

**문서 템플릿:**

```markdown
# {주제명} 합의 문서

> 작성일: {날짜}

## 배경

{왜 이 합의가 필요했는지 한 문장으로}

## 합의 결과

| 주제 | 결정 내용 | 근거 |
|------|---------|------|
| {주제 1} | {결정} | {이유} |
| {주제 2} | {결정} | {이유} |

## 미결 사항

- {구현 중 결정해도 되는 사항} (없으면 생략)

## 다음 단계

{사용자가 선택한 다음 스킬}
```

#### 3-3. 다음 단계 선택 안내

**⚠️ 절대 규칙: 문서 저장 후 반드시 아래 선택지를 출력하고 사용자 응답을 기다린다.**
**어떤 상황에서도 AI가 임의로 다음 스킬을 실행하거나 단계를 건너뛰면 안 된다.**
**"진행해줘", "계속해줘" 같은 모호한 답변이 와도 반드시 A/B/C 중 하나를 명확히 선택하게 한다.**

```
## 합의 완료 ✅

docs/.consensus/{주제명}.md 저장됨.

다음 단계를 선택해주세요:

**A. /domain-definition** ← 추천 (신규 도메인이거나 Aggregate 경계가 복잡한 경우)
도메인 모델(Entity, Repository, 비즈니스 규칙 등)을 먼저 정의하고 계획을 세웁니다.

**B. /writing-plan** (기존 도메인에 기능 추가이거나 단순 구현인 경우)
도메인 정의 없이 바로 구현 계획을 작성합니다.

**C. 종료** (합의까지만 진행)
계획과 도메인 정의는 나중에 작성합니다.
```

추천 기준:
- 새 테이블/엔티티 생성, Aggregate 새로 정의 → **A 추천**
- 기존 도메인에 필드/API 추가, 단순 CRUD 수정 → **B 추천**

사용자가 **A**를 선택하면 `/domain-definition` 스킬을 실행한다.
사용자가 **B**를 선택하면 `/writing-plan` 스킬을 실행한다.
사용자가 **C**이거나 "나중에"라고 하면 종료한다.

**절대 금지:**
- 사용자 선택 없이 다음 스킬 자동 실행
- A/B/C 선택지 제시 생략
- "진행해줘"를 B 선택으로 임의 해석하여 /writing-plan 바로 실행
- domain-definition 단계 무언급 스킵

## 언제 합의를 종료하는가

다음 조건이 모두 충족되면 합의 완료:

1. 핵심 목적이 명확히 정의됨
2. 구현 범위가 합의됨
3. 주요 기술적 선택이 결정됨
4. 우선순위 모호함이 해소됨

5개 이상의 질문이 필요하다면, 중간에 "지금까지 파악한 내용"을 정리하고 나머지 모호함을 확인하라. 지나치게 많은 질문은 사용자를 피로하게 만든다.

---

## 예시: 질문 패턴

### 좋은 예시

```
## 1. 캐시 적용 레이어

이 조회 API는 빈번히 호출되므로 캐시 위치에 따라 성능과 일관성이 달라집니다.

**선택지:**
- **A. Application 레이어 (Caffeine)** — 로컬 인메모리, 응답 가장 빠름, 서버 재시작 시 캐시 소멸 ← 추천
- **B. Redis** — 분산 캐시, 서버 재시작 후에도 유지, 네트워크 비용 존재
- **C. 캐시 없음** — 단순하지만 DB 부하 증가

**추천: A (Caffeine)**
현재 이 데이터는 1분에 한 번 정도만 변경되고, 서비스 단위 배포라 캐시 무효화가 간단합니다. Redis를 쓸 만큼 복잡한 분산 환경이 아직 아닙니다.
```

### 나쁜 예시 (이렇게 하지 마라)

```
캐시를 어디에 적용할지 생각해보셨나요? Redis나 Caffeine 모두 좋은 옵션입니다.
어떤 방향이 좋으실 것 같으세요?
```

## 컨텍스트 활용

코드베이스가 있는 경우:
- 관련 파일 먼저 탐색 후 현재 패턴 파악
- 기존 코드와 일관된 방향으로 추천 제시
- "이 프로젝트에서는 이미 X 패턴을 사용하고 있으므로 A를 추천합니다"처럼 근거 강화

## 출력 언어

**항상 한글로 작성한다.** 기술 용어(API, DB, Redis 등)는 그대로 사용해도 된다.
