# Implementation 스킬

## 역할

작성된 계획을 TDD 방식으로 구현한다. 테스트를 먼저 작성하고(Red), 통과시키고(Green), 개선하는(Refactor) 사이클을 태스크마다 반복한다. 이 순서를 지키는 이유는 구현 전에 "무엇을 만들어야 하는가"를 테스트로 명확히 정의하기 위해서다.

---

## Phase 1: 계획 로드

`docs/.plan/` 디렉토리를 탐색하여 계획 파일을 읽는다.

- 파일이 하나면 그것을 사용
- 여러 개면 파일 목록을 보여주고 선택 요청
- 없으면 `$ARGUMENTS`에서 구현할 내용 파악. 그래도 없으면 직접 질문

계획에서 추출할 항목:
- 태스크 목록과 순서
- 각 태스크의 대상 파일, 구현 내용, 완료 기준
- 관련 비즈니스 규칙 (domain-definition 문서가 있으면 함께 읽기)

---

## Phase 2: 모드 선택

구현 시작 전에 사용자가 모드를 선택한다.

```
구현 모드를 선택해주세요:

**A. Learning 모드** — 직접 구현하며 배우기
AI가 실패하는 테스트를 작성하면, 사용자가 직접 로직을 구현하여 테스트를 통과시킵니다.
코드를 직접 작성하면서 설계 의도를 이해하고 싶을 때 선택하세요.

**B. Subagent 모드** ← 추천 (컨텍스트 오염 없음)
태스크마다 독립 subagent가 TDD 사이클을 완료하고 결과만 보고합니다.
현재 세션 컨텍스트를 깨끗하게 유지하면서 AI가 구현하게 하고 싶을 때 선택하세요.

**C. In-process 모드** — 현재 세션에서 직접 구현
AI가 현재 세션에서 바로 TDD 사이클을 진행합니다.
빠르게 진행하거나 과정을 직접 관찰하고 싶을 때 선택하세요.
```

---

## TDD 사이클 (3단계 공통)

모드에 관계없이 **태스크마다 아래 사이클을 반복**한다.

### 🔴 RED: 실패하는 테스트 먼저

구현 전에 테스트를 작성하고 실패를 확인한다. 테스트가 실패해야 하는 이유는 아직 구현이 없기 때문이다. 이 단계에서 테스트가 통과하면 테스트가 잘못된 것이다.

- 태스크의 완료 기준을 테스트 코드로 표현
- 프로젝트 테스트 프레임워크 사용 (Kotest, JUnit 등 CLAUDE.md 참조)
- 테스트 실행하여 실패 확인
- 실패 메시지를 기록 (어떤 이유로 실패하는지 명확히)

### 🟢 GREEN: 테스트를 통과시키는 최소 구현

테스트가 통과하는 가장 단순한 구현을 작성한다. 이 단계에서 목표는 "완벽한 코드"가 아니라 "통과하는 코드"다. 과도한 최적화나 추상화는 Refactor 단계로 미룬다.

- 테스트가 요구하는 것만 구현
- 테스트 실행하여 통과 확인
- 다른 기존 테스트가 깨지지 않았는지 확인

### 🔵 REFACTOR: 코드 개선

동작은 유지하면서 코드 품질을 개선한다. 테스트가 있기 때문에 리팩터링이 안전하다.

- 중복 제거, 네이밍 개선, 추상화 수준 조정
- 프로젝트 코딩 컨벤션 적용
- 개선 후 테스트 재실행하여 여전히 통과하는지 확인
- 개선 사항이 없으면 생략 가능

## 모드별 실행 방법

### A. Learning 모드

사용자가 직접 구현하며 배우는 모드. AI는 테스트와 힌트를 제공하고, 구현은 사용자가 한다.

**태스크 진행 흐름:**

1. **RED 단계**: AI가 실패하는 테스트 코드 작성 → 사용자에게 전달
   ```
   ## Task {N}: {태스크명}

   🔴 아래 테스트를 {경로}에 작성했습니다. 실행해서 실패를 확인하세요:

   [테스트 코드]

   **구현 힌트:**
   - {무엇을 구현해야 하는지 방향 제시 — 코드는 주지 않음}
   - {핵심 비즈니스 규칙 언급}

   구현 완료 후 테스트를 실행하고 결과를 알려주세요.
   ```

2. **GREEN 단계**: 사용자가 구현 → 테스트 실행 결과 보고 → AI가 확인
   - 통과: 다음 단계로
   - 실패: 추가 힌트 제공 (코드는 여전히 주지 않음)
   - 사용자가 포기하면: 정답 코드 제공 후 설명

3. **REFACTOR 단계**: AI가 리팩터링 포인트 제안 → 사용자가 직접 개선

---

### B. Subagent 모드

태스크마다 독립 subagent를 생성하여 TDD 사이클을 위임한다. 현재 세션은 오케스트레이터 역할만 하며 컨텍스트를 보존한다.

**태스크 진행 흐름:**

1. 계획에서 태스크 목록을 전부 추출
2. 태스크 순서대로 subagent 생성:

   **Subagent 프롬프트 구조:**
   ```
   다음 태스크를 TDD(Red→Green→Refactor) 방식으로 구현하라.

   [태스크 전체 내용 — 파일 경로, 구현 내용, 완료 기준]
   [관련 비즈니스 규칙]
   [프로젝트 컨벤션 요약]

   실행 순서:
   1. 실패하는 테스트 먼저 작성하고 실패 확인
   2. 테스트를 통과하는 최소 구현
   3. 리팩터링 (필요 시)
   4. 최종 테스트 통과 확인

   완료 후 보고:
   - 생성/수정한 파일 목록
   - 테스트 결과
   - 특이사항 또는 결정 사항
   ```

3. Subagent 완료 보고 수신 → 현재 세션에 요약만 출력:
   ```
   ✅ Task {N} 완료: {태스크명}
   - 변경 파일: {목록}
   - 테스트: {통과 수}/{전체 수}
   - 특이사항: {있으면 표시}
   ```

4. 모든 태스크 완료 후 전체 요약 출력

**Subagent 상태 처리:**

| 상태 | 대응 |
|------|------|
| 완료 | 다음 태스크 진행 |
| 테스트 실패 | 실패 내용 확인 후 재시도 또는 에스컬레이션 |
| 블로커 발생 | 사용자에게 보고 후 결정 대기 |
| 컨텍스트 부족 | 추가 정보 포함하여 재위임 |

---

### C. In-process 모드

현재 세션에서 AI가 직접 TDD 사이클을 진행한다.

**태스크 진행 흐름:**

1. 태스크 시작 선언:
   ```
   ## Task {N}/{전체}: {태스크명}
   ```

2. 🔴 RED 실행:
   - 테스트 코드 작성
   - 테스트 실행 → 실패 확인
   - 실패 메시지 출력

3. 🟢 GREEN 실행:
   - 최소 구현 코드 작성
   - 테스트 실행 → 통과 확인
   - 기존 테스트 깨짐 여부 확인

4. 🔵 REFACTOR 실행:
   - 개선 사항 있으면 적용
   - 테스트 재실행

5. 태스크 완료 선언 후 다음 태스크로

## 공통 규칙

**테스트 실패 대응:**
- 테스트가 예상과 다른 이유로 실패하면 → 테스트 재검토 후 수정
- 구현이 복잡하여 단계가 막히면 → 더 작은 단위로 분리

**중단 조건 (모든 모드 공통):**
- 의존하는 외부 시스템이 없거나 미구현 상태
- 계획에 누락된 컨텍스트가 있어 구현 방향이 불명확
- 연속 3회 이상 같은 테스트 실패

중단 시 사용자에게 상황을 보고하고 결정을 요청한다. 임의로 계획을 변경하지 않는다.

**완료 보고 (모든 태스크 완료 후):**
```
## 구현 완료 ✅

모드: {선택한 모드}
완료 태스크: {N}/{전체}
변경 파일: {목록}
테스트 결과: {통과}/{전체}

다음 단계를 선택해주세요:

**A. /implementation-review** ← 추천
계획/합의/도메인 문서 대비 구현 적합성을 점수 기준으로 자동 검증합니다.

**B. /implementation-docs**
구현 가이드 문서를 docs/ 하위에 작성합니다. (검증 없이 바로 문서화)

**C. 종료** (검증/문서화는 나중에)
```

## 출력 언어

**항상 한글로 작성한다.** 코드, 파일 경로, 클래스명, 테스트 실행 명령어는 원어 그대로 사용한다.
