# Multi-Agent 코드 리뷰

두 AI(Codex, Gemini)가 코드를 독립적으로 리뷰하고, 리드(Claude)가 통합하여 최종 판정한다.

---

## Step 1: 리뷰 대상 파악

리뷰를 시작하기 전에 다음을 확인한다.

### 리뷰 모드 선택

| 모드 | 언제 사용 | REVIEW_TARGET |
|------|----------|---------------|
| `uncommitted` | 현재 작업 중인 미커밋 변경사항 | (없음) |
| `branch` | 특정 브랜치 대비 변경사항 | 브랜치명 (예: `main`) |
| `diff_file` | 사용자가 diff 텍스트를 붙여넣거나 파일 경로를 제공 | diff 파일 경로 |

사용자 입력에서 모드를 판단한다:
- diff 텍스트를 직접 붙여넣었다면: `/tmp/team-review-diff.txt`에 저장 후 `diff_file` 모드
- 파일 경로를 제공했다면: 해당 경로를 그대로 사용하여 `diff_file` 모드
- "미커밋", "uncommitted", "현재 변경사항"이면: `uncommitted` 모드
- "main 대비", "브랜치 비교" 등이면: `branch` 모드 + 브랜치명

모드가 불분명하면 사용자에게 한 번 확인한다. 이후 바로 에이전트를 병렬 실행한다.

### diff 텍스트 저장 (diff_file 모드에서 붙여넣기인 경우)

```bash
cat > /tmp/team-review-diff.txt << 'DIFF_EOF'
{사용자가 붙여넣은 diff 내용}
DIFF_EOF
```

## Step 2: 병렬 에이전트 실행

**반드시 두 에이전트를 동시에 호출한다.** 하나의 응답에서 Agent 도구를 두 번 호출하여 병렬로 실행한다.

각 에이전트에게 아래 정보를 프롬프트로 전달한다:
- `agents/codex-reviewer.md` 내용 (Read로 먼저 읽은 후 포함)
- `agents/gemini-reviewer.md` 내용 (Read로 먼저 읽은 후 포함)
- REVIEW_MODE와 REVIEW_TARGET 값

### Agent1 — Codex 리뷰어

```
다음 지시에 따라 Codex CLI로 코드 리뷰를 수행하라.

{agents/codex-reviewer.md 전체 내용}

---
REVIEW_MODE: {uncommitted | branch | diff_file}
REVIEW_TARGET: {브랜치명 또는 diff 파일 경로}
---

CLI 실행 후 결과를 지정된 형식으로 반환하라.
CLI 실행 시 타임아웃은 300초(5분)로 설정한다.
```

### Agent2 — Gemini 리뷰어

```
다음 지시에 따라 Gemini CLI로 코드 리뷰를 수행하라.

{agents/gemini-reviewer.md 전체 내용}

---
REVIEW_MODE: {uncommitted | branch | diff_file}
REVIEW_TARGET: {브랜치명 또는 diff 파일 경로}
---

CLI 실행 후 결과를 지정된 형식으로 반환하라.
CLI 실행 시 타임아웃은 300초(5분)로 설정한다.
```

## Step 3: 리드 에이전트 통합 판정

두 에이전트 결과를 받은 후, 리드(Claude 자신)가 아래 기준으로 통합 판정한다.

### 에이전트 결과 상태 확인

먼저 각 에이전트의 실행 상태를 확인한다:

- **양쪽 성공**: 정상 통합 판정 진행
- **한쪽 실패**: 성공한 에이전트의 결과만으로 리뷰 진행, 출력에 실패 사실을 명시
- **양쪽 실패**: 리드가 직접 Claude 기반 리뷰를 수행하고, "외부 에이전트 실행 불가" 명시

### 이슈 교차 검증

각 이슈를 파일 경로와 내용 기준으로 대조한다:

| 상황 | 처리 방법 |
|------|----------|
| **양쪽 모두 지적** | 채택. 신뢰도 높음. `근거: Codex ✅ / Gemini ✅` 표시 |
| **한쪽만 지적** | 리드가 직접 판단. 타당하면 채택, 아니면 기각. 채택 시 근거 명시 |
| **양쪽 의견 충돌** | 리드가 이유를 설명하고 최종 판정. 불일치 테이블에 별도 표시 |
| **리드가 추가 발견** | 두 에이전트 모두 놓친 이슈를 리드가 직접 추가. `근거: Lead 직접 판단` 표시 |

### Pn 우선순위 부여

이슈 심각도를 다음 기준으로 Pn 레벨로 변환한다:

| Pn | 기준 | 대응 |
|----|------|------|
| **P1** | 버그·보안·데이터 손상 위험 / Critical | 반드시 수정 |
| **P2** | 품질·유지보수성·일관성 / Major | 적극 반영 권고 |
| **P3** | 개선 아이디어·선호 패턴 / Minor | 가능하면 반영 |
| **P4** | 대안 제시·선택적 개선 / Nitpick | 응답 불필요 |
| **P5** | 스타일·가독성·취향 | 무시 가능 |

### 중복 제거

같은 파일·라인을 다르게 표현한 이슈는 하나로 합친다. 더 구체적인 설명을 채택한다.

---

## Step 4: 최종 리뷰 출력

```markdown
## 코드 리뷰 (Multi-Agent)

> **리뷰 방식**: Codex CLI + Gemini CLI 병렬 리뷰 → 리드 에이전트 통합 판정
> **리뷰 기준**: Pn 룰 (P1 필수 반영 ~ P5 사소한 의견)
> **에이전트 상태**: Codex {✅ 성공 | ❌ 실패} / Gemini {✅ 성공 | ❌ 실패}

---

### {파일명 또는 변경 영역}

**P1** `파일경로:라인번호` — 이슈 제목
> 왜 문제인지 설명한다.
> 어떤 방향으로 수정하면 좋은지 안내한다.
> 근거: Codex ✅ / Gemini ✅

**P2** `파일경로:라인번호` — 이슈 제목
> 설명.
> 수정 방향.
> 근거: Codex ✅ / Gemini ❌ (리드 판단으로 채택 — {이유})

---

### 에이전트 간 의견 불일치

(불일치 항목이 없으면 이 섹션은 생략한다)

| 항목 | Codex 의견 | Gemini 의견 | 리드 판정 |
|------|-----------|-------------|----------|
| `파일:라인` — {이슈} | {Codex 입장} | {Gemini 입장} | {리드 결정 및 이유} |

---

### 요약

| 레벨 | 건수 | 주요 내용 |
|------|------|---------|
| P1 (필수) | N | 항목 나열 |
| P2 (권고) | N | 항목 나열 |
| P3 이하 | N | 항목 나열 |
```

### 코멘트 작성 원칙

- **이유를 함께 쓴다** — "수정하세요"가 아니라 왜 문제인지를 먼저 설명한다
- **수정 방향을 제시한다** — 정답을 주기보다 어떤 방향으로 개선할지 안내한다
- **코드가 아닌 행동에 대한 것** — 작성자를 비판하지 않는다
- **개인 취향은 P4/P5로** — 팀 규약이 아닌 선호는 낮은 우선순위로 표시한다

## 정리

리뷰 완료 후 임시 파일이 생성된 경우 정리한다:

```bash
rm -f /tmp/team-review-diff.txt
```
